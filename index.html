What this is

A from-scratch recreation by Jetha Chan (@jetha) using Grok 4.5, driven by Abby’s public footage, her technique write-up, and iterative visual fixes. Loop: cocoon → mid → moth → full noisy center-to-edge wipe (moth design rotates each cycle). Animation lives in the fragment shader (iTime / iResolution only — Shadertoy-style).

How to poke it

Open the Shader tab. Edit the GLSL, hit Compile (or Ctrl+Enter). Errors show under the editor without killing the last good program.

Turn 1

Brief: rebuild the spinner from Abby’s tweet + video

Pulled Abby’s post and the attached clip. Goal: a procedural WebGL recreation using SDFs, bit-packed stages, and per-cell on/off from position + shape + time + noise — plus a simple tester page.

Sampled the video into a 21×21 grid, extracted cocoon / mid / moth bitmasks, built rounded-rect cells and registration marks, and stood up a first shader host. Early animation “worked” but wasn’t faithful yet.

Turn 2

Radial ripple, not a top-down scan

First lime pass was a vertical scan band. Wrong. Frame analysis showed lime mean radius growing from center (~6 cells/s, band width ~2.3). Rewrote form / dissolve as a center-out ring.

Turn 3

Arrival model + lime past the glyph

Clarified the intended model: new pixels arrive at the center and ripple out to form the pattern. Lime rides that front on the silhouette and can leak past the glyph, losing energy toward empty grid / boundary — not a free disc, not a hard mask.

Restructured toward a Shadertoy style: almost everything in the fragment shader; JS only feeds time and resolution.

Turn 4

Shadertoy host: iTime only, live editor

Split shader.frag from a thin main.js host. Sidebar UI with About + live Shader tab (debounced recompile, keep last good program on error). Timeline lives entirely in GLSL.

Turn 5

Falloff, wipes, and “start empty”

Iterated lime falloff (too soft → too hard → middle ground). Removed invented corner / edge wipe sparkles that looked “registration-y” but weren’t right. Stopped opening mid-hold on a full moth — t = 0 is empty, then cocoon forms from the center.

End of loop became a full noisy center-to-edge wipe that clears the whole grid, not only the silhouette.

Turn 6

Credits, shipping, Pages + Cloudflare DNS/HTTPS

Explicit credit: reconstruction by Jetha Chan (@jetha) driving Grok 4.5, of Abby Welsh’s (@DubThinkDev) work — not shipping source. Footer links to both X profiles and grok.com; tweet embed in About.

Shipped MIT on GitHub, GitHub Pages, custom domain marathon-spinner.jethachan.net. Used Bitwarden Secrets CLI (bws) for the Cloudflare API token, created the CNAME, proxied for TLS, zone Always HTTPS / HSTS.

Turn 7

Stage order: cocoon → mid → moth → wipe

Bitmasks already matched sampled holds (majority vote vs video). The “slightly wrong” feel was mostly sequence: morphing moth→moth skipped the mid wing-sprout stage. Restored footage order and chubbier cells; center lime “heart” on holds.

Turn 8 · latest

Jetha reports a visual artifact; Grok one-shots the fix

On phone, the rightmost five columns looked horizontally scrambled. Desktops were fine.

Diagnosis: mobile fragment shaders often default int/uint to mediump (~16 bits). A 21-wide grid stores columns 16–20 in bits past 16 — exactly those five columns. Classic GLSL ES trap.

Fix in one pass: force precision highp int, and split each row mask into a uvec2 lo/hi pair (cols 0–15 / 16–20) so shifts never leave the 16-bit-safe range. Also a tiny epsilon on cell floor(). Shipped.
